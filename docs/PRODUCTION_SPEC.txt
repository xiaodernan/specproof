======================================================================
0. 安全前置
======================================================================

1. 任何已粘贴到聊天、Issue、PR、日志或公开页面中的 API Key 都必须立即轮换。
2. 本项目绝不在代码、README、Prompt、测试快照或前端中保存真实密钥。
3. 所有密钥只从环境变量、Docker Secret 或生产 Secret Manager 读取。
4. DeepSeek、GitHub App、数据库和对象存储密钥相互隔离。
5. Agent 永远不能读取宿主机环境变量、Docker Socket 或仓库 Secrets。
6. 示例中仅使用占位符：
   LLM_API_KEY=replace_me

======================================================================
1. 三句话说清产品
======================================================================

1. 用户把需求、验收条件和一个 AI 生成的 GitHub PR 交给 SpecProof。
2. SpecProof 由独立验证 Agent 把需求编译为可执行 Contract，并在修改前后的两个版本上运行 API、数据库、Redis、RabbitMQ、静态分析、差分测试、属性测试和有限变异测试，主动寻找反例。
3. 只有每条关键验收条件都有可重放证据时，系统才签发 Merge Certificate；否则返回失败测试、最小反例、影响路径和一键重放的 Bug Capsule。

一句话：
“实现 Agent 不能给自己签字；SpecProof 用可执行证据证明一个 AI PR 是否真的完成需求。”

产品类别：
AI Change Acceptance Firewall
不是普通 Code Review Bot。

======================================================================

设计目标评分：

1. 面试工程含量：9.8 / 10
   - Agent、RAG、工具调用、状态机、差分测试、变异测试、消息可靠性、

2. 单兵演示效果：9.6 / 10
   - 一条 CLI 命令或一个 GitHub PR 即可展示。
   - 不需要真实流量、线上事故或多台服务器。
   - 现场植入一个鉴权或事务 Bug，即可生成失败测试和证据包。

3. 产品原创性：9.1 / 10
   - 核心不是“AI 给代码提意见”，而是“需求到证据的独立验收”。
   - 高等级 Finding 必须有可执行证据。
   - 输出 Requirement-to-Evidence Matrix、Bug Capsule 和签名证书。

4. 细分市场商业优势：9.0 / 10
   - 只与“AI 生成的 Spring 后端 PR 验收”这一窄场景比较。
   - 私有部署、BYOK、后端语义验证和可重放证据是主要优势。
   - 该分数必须通过第 29 节量化指标后才正式成立。

======================================================================
4. 与普通 AI Code Review 的本质差异
======================================================================

普通 AI Review：
- 输入主要是 Diff 和仓库上下文。
- 输出主要是评论、风险和修复建议。
- 很多结论依赖模型判断。
- 测试常由实现 Agent 自己生成。
- “测试通过”不等于需求完成。

SpecProof：
- 输入是 Requirement + Acceptance Criteria + Forbidden Changes + PR。
- 输出是每一条需求对应的可执行证据。
- 严重问题必须能够重放。
- 验证 Agent 与实现 Agent 隔离。
- 同时运行 Base 与 Head，识别 PR 真正引入的变化。
- 比较 HTTP、MySQL、Redis、RabbitMQ 和性能语义。
- 证据绑定 Commit SHA、需求版本、工具版本和运行环境。
- 通过后签发机器可验证的 Merge Certificate。

产品不回答：
“这段代码看起来怎么样？”

产品回答：
“它是否完成了任务、破坏了什么，以及证据在哪里？”

5. 核心创新护城河
======================================================================

5.1 Requirement-to-Evidence Matrix

示例：

验收条件：
- 未登录用户不得修改邮箱。
- 邮箱必须唯一。
- 修改邮箱后旧 Token 失效。
- 原有 API 保持兼容。
- 事件只投递一次。

系统输出：

| Contract | Changed Symbols | Experiment | Result | Evidence |
|----------|-----------------|------------|--------|----------|
| AUTH-01  | Controller      | HTTP auth test | FAIL | Capsule A |
| USER-07  | Service/DB      | duplicate email test | PASS | Run B |
| TOKEN-03 | TokenService    | base/head differential | FAIL | Capsule C |
| API-02   | OpenAPI         | schema diff | PASS | Report D |
| MQ-04    | Publisher       | redelivery test | PASS | Trace E |

5.2 Independent Verifier

实现代码的 Agent 不参与签发结果。
SpecProof 只读取：
- 原始需求
- 用户批准的 Contract
- Base SHA
- Head SHA
- 可执行环境
- 工具输出

它不相信：
- 实现 Agent 的自我评价
- 实现 Agent 生成的“全部通过”摘要
- PR 描述中未经验证的声明

5.3 Proof-Carrying Finding

BLOCKER / MAJOR 至少包含一种强证据：
- Base 通过、Head 失败的测试
- 可重放最小反例
- 可达静态数据流
- Contract 的确定性失败
- API / Schema / Migration 的确定性破坏
- Redis / MQ 状态机失败
- 有效变异体与能杀死它的新测试
- Verifier 实际重放成功

5.4 Semantic Differential Execution

同一输入分别运行 Base 和 Head，比较：
- HTTP status / body / headers
- MySQL 行变化与约束
- Redis Key / TTL / 失效行为
- RabbitMQ 消息数量、顺序、Routing Key、Payload 和重复投递
- 异常类型
- 外部调用数量
- SQL 查询数量
- 延迟和内存
- OpenAPI Schema

Agent 负责判断差异是：
- EXPECTED_CHANGE
- REGRESSION
- AMBIGUOUS
- NON_REPRODUCIBLE

5.5 Mutation-Guided Test Synthesis

只对 PR 影响范围做变异测试：
- 条件边界
- 返回值
- 空值检查
- 权限判断
- 事务处理
- 重试和幂等
- 消息 Ack
- Redis 失效逻辑

存活变异体不自动视为 Bug。
Agent 先判断是否可能等价，再生成最小测试。
只有实际执行成功，才成为 Test Weakness 或 Finding。

5.6 Repository Constitution

从以下来源提取候选 Contract：
- Issue / Ticket
- ADR
- README
- API 文档
- 数据库约束
- 现有测试
- 相似历史实现
- 安全规范

候选规则必须经过用户批准。
批准后尽量编译成：
- Semgrep
- AST Query
- ArchUnit
- OpenAPI Rule
- SQL / Migration Check
- JUnit Test Template
- Redis / MQ Scenario
- JSON Schema

5.7 Bug Capsule

每个确认问题生成可移植证据包：

capsule/
├── manifest.json
├── requirement.json
├── contract.json
├── finding.json
├── generated-tests/
├── fixtures/
├── base-result/
├── head-result/
├── tool-reports/
├── environment/
├── run.sh
└── README.md

在另一台机器运行：

    specproof replay capsule.zip

应得到同样结果。

5.8 Signed Merge Certificate

通过后生成：

{
  "subject": {
    "repository": "...",
    "commit_sha": "..."
  },
  "requirements_digest": "sha256:...",
  "contracts_digest": "sha256:...",
  "result": "VERIFIED",
  "verified_contracts": 12,
  "unverified_contracts": 0,
  "evidence_digests": [],
  "toolchain": {},
  "environment_digest": "sha256:...",
  "issued_at": "...",
  "issuer": "SpecProof"
}

证书：
- 使用 in-toto Statement 风格。
- 绑定 Commit SHA 和需求版本。
- 可使用 Ed25519 签名。
- 可选发布为 GitHub Artifact Attestation。
- 可作为 GitHub Required Check。

FAST（目标 2～5 分钟）：
- 需求编译
- 静态分析
- API Schema
- 定向单测
- 关键 Contract
- 有限差分测试

- Stateful API Fuzzing
- MySQL / Redis / MQ 差分
- 影响范围变异测试
- 属性测试
- Review Court

RELEASE（目标 10～30 分钟）：
- 全部关键 Contract
- 扩大测试范围
- Capsule 重放
- 证据哈希
- 签名 Certificate
- 发布 Required Check

风险调度器根据：
- 改动规模
- 涉及权限
- Migration
- Redis / MQ
- 公开 API
- 历史事故
- 业务 Contract
选择验证档位和预算。

10. Review Court
======================================================================

Prosecutor：
- 提出潜在违规
- 指明影响路径
- 设计所需证据

Defender：
- 搜索反证
- 检查不可达路径
- 查找已有保护
- 识别预期行为
- 识别等价变异体

Experiment Designer：
- 选择静态证明、属性测试、差分测试、状态机测试、变异测试或性能测试

Verifier：
- 实际执行
- 检查 Exit Code
- 确认 Base / Head
- 校验证据哈希
- 确认问题由 PR 引入
- 确认代码行
- 去重

Judge：
- 只依据 Evidence Policy
- 不依据谁的自然语言更像专家

发布门槛：

BLOCKER：
- 强可执行证据
- replay 成功
- confidence >= 0.90

MAJOR：
- 至少一种强证据
- confidence >= 0.82

MINOR：
- 仓库证据 + 逻辑验证
- confidence >= 0.72

NEEDS_CONFIRMATION：
- 合理怀疑，但无法稳定验证
- 不阻止合并

12. Sandbox 威胁模型
======================================================================

PR 代码是不可信输入。

必须：
- 非 root
- 默认断网
- 只读源代码
- 独立可写 tmp
- CPU 限制
- RAM 限制
- PID 限制
- 磁盘限制
- Timeout
- 不挂 Docker Socket
- 不传 LLM Key
- 不传 GitHub App 私钥
- Secret Redaction
- 工具输出截断
- Fork PR 更严格

依赖安装：
- 优先使用 Lock File 和预构建镜像。
- 默认禁止任意 install hook。
- Allowlist 后才能执行脚本。

Prompt Injection：
- README、源码、Issue 和 PR 文本全部是数据。
- 它们不能改变系统策略或工具权限。
- Tool Gateway 独立鉴权。

24. 最终面试叙事
======================================================================

“现在 Coding Agent 写代码越来越快，问题不再是缺少第二个会评论代码的模型，
而是实现者不能给自己签发合格证。

我做的 SpecProof 接收原始需求和 PR，把需求转为经过人工批准的 Contract，
然后在 Base 与 Head 两个隔离环境中对 HTTP、MySQL、Redis 和 RabbitMQ 行为做差分验证。
高等级问题必须附带失败测试或可重放反例。

系统使用 LangGraph 控制验证流程；
MySQL 保存业务事实；
MongoDB 保存复杂实验工件；
Elasticsearch 做仓库级混合检索；
Redis 做锁、缓存和实时进度；
RabbitMQ 承载可靠长任务；
MinIO 保存证据包和证书；
Outbox 与幂等保证任务不丢不重；
最后用签名 Certificate 绑定需求版本、Commit 和证据。

它不是告诉我代码可能有问题，而是证明这个 AI PR 是否真正完成了任务。”
